
liboqs Optimizations:

1. Faster compilation
A standard liboqs build spends a lot of time compiling automated tests, benchmarking tools, and example programs. Because we only want the library itself to use in your Python or C wrappers, compiling those extra executables is a waste of time. This flag tells CMake to skip them entirely. Combined with the minimal build flag, this reduces our liboqs compile time from minutes down to just a few seconds.



Wrapper Optimizations:

1. Getting Accurate Timing Measurements
Testing cryptographic algorithms requires highly exact timing. If we measured the execution time in Python, the results would be inaccurate because they would include the delay caused by Python's interpreter and the time it takes to move data between Python and C(liboqs). To get an exact measurement, the C wrapper uses native C timing functions (clock_gettime). This measures the exact time the processor spends running the cryptography algorithms natively, completely ignoring any delays caused by Python.

2. Reducing the Python-to-C Overhead
Moving data back and forth between Python and C takes time. If Python called the liboqs library directly, it would have to cross this boundary multiple times just to finish one test-once to set up memory, once to make the keys, once to encrypt, and once to clean up memory. The C wrapper solves this by grouping all these steps into a single function. Python only has to call the wrapper once, which cuts down on overhead and makes the whole testing process run faster.
